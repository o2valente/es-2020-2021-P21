Descrição da Aplicação


A Aplicação tem como objetivo monitorizar a atividade de um soldado, mostrando a sua localização em tempo real e os seus dados vitais. Os dados de monitorização são captados por GPS, pelo capacete do soldado e pelo seu dispositivo de batimentos cardíacos.

Na página Map é possível ver o soldado a movimentar-se no terreno,na página GPS é apresentado os dados captados pelo dispositivo GPS do soldado, na página Helmet podemos ver os dados recolhidos pelo capacete, e na página do ECG teremos a atividade cardíaca.

À medida que os dados do soldado são recebidos, se for observada atividade fora do normal, como a pulsação ou níveis elevados de Monóxido de Carbono nos pulmões, são enviadas mensagens de alerta/perigo para a página Map,para que quem estiver a monitorizar o percurso do soldado conseguir enviar reforços a tempo.



Features

Mapa em tempo real

Mapa com a posição do soldado em tempo real.

GPS

Verificar dados precisos relativos à posição do soldado no mapa, como longitude, latitude e altitude.

Monitorização

Consulta fácil de dados vitais sobre o soldado, como os níveis de Monóxido de Carbono e batimentos cardíacos.

Alertas

Alertas em tempo real caso alguns dos dados vitais que estam a ser monitorizados estejam fora do normal.

Perspetiva da Operação



Deploy - Jenkins Pipeline


Para a definição do pipeline Jenkins, inicia-se guardando os dados da VM onde será feito o deploy de algumas componentes da aplicação. O primeiro stage é a clonagem do repositório Github, de forma a obter a última versão do código desenvolvido. Após esta etapa, é realizada uma compilação através de Maven (mvn clean install) deste mesmo código, seguidos dos testes (unitários e de integração) desenvolvidos, através também de comandos Maven (mvn test). Depois de verificar a estabilidade do código do projeto, faz-se o deploy do JAR gerado para a VM de Artifactory, de forma a guardar todas as versões de desenvolvimento do projeto. Seguidamente, dá-se lugar ao build da aplicação Spring e da tecnologia React associada para imagens docker que, por sua vez, serão atualizadas no Docker Registry fornecido. Finalmente, é realizado o deploy das componentes do projeto que possam mudar entre builds: é estabelecida uma conexão com a Virtual Machine de Playground, através das credenciais específicas para o nosso grupo, e atualiza-se a mesma com novos containers, correndo novas imagens docker das componentes do nosso projeto. Este processo consiste na paragem e remoção dos containers antigos, a obtenção das novas imagens do docker registry, e a criação e início dos containers com a última versão do projeto.

Considerações finais



Dificuldades


Os testes Cucumber revelaram-se uma grande dificuldade, não pelo conceito ou implementação do código dos mesmos, mas pela sua integração com os testes feitos posteriormente. Tendo esta integração não ter sido possível.

Sendo a maior parte das tecnologias desconhecidas para os membros do grupo, foi preciso trabalho extra de forma a poder implementar as milestones com sucesso.



O que mudaríamos?


Os dados fornecidos não eram os melhores, talvez por serem ficheiros ao invés de dados in real time provenientes de uma API, o que consequentemente leva a que o tema em si fique um pouco menos interessante.
